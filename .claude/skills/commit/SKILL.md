# commit 작성 커맨드

이 커맨드는 현재 워킹 디렉토리의 변경사항을 파악하여 커밋 메시지를 제안하고, 사용자의 확인을 받은 후 커밋 및 푸시를 수행합니다.

## 이 커맨드가 하는 일

1. **변경사항 확인** - 코드의 변경사항을 명확히 확인

- `git status` 명령어를 호출하여 변경된 파일 목록 확인
- `git diff` 명령어를 호출하여 unstaged 변경사항 확인
- `git diff --staged` 명령어를 호출하여 이미 staged된 변경사항 확인
- `git log -n 5 --oneline` 명령어를 호출하여 최근 커밋 메시지 스타일(컨벤션) 파악

2. **커밋 대상 및 커밋 메시지 제안** - 확인된 변경사항을 바탕으로 커밋 메시지 초안 작성

- 어떤 파일을 `git add`할지 목록으로 정리하여 제시
- 변경사항에 맞는 커밋 메시지를 아래의 형식에 맞춰 작성
- **[매우 중요]** 제안 내용을 사용자에게 출력한 후 반드시 승인을 요청하고 **대기**합니다. 승인 없이는 절대로 add, commit, push를 수행하지 않습니다.

3. **커밋 및 푸시 수행** - 사용자가 승인한 경우에만 실행

- 사용자가 "네", "좋아요", "진행해주세요", "ok", "ㄱㄱ" 등 긍정적인 응답을 한 경우에만 다음 단계를 진행합니다.
- 제안한 파일들을 `git add`합니다.
- 승인된 커밋 메시지로 `git commit -m "<메시지>"`를 실행합니다.
- `git branch --show-current` 명령어로 현재 브랜치를 확인합니다.
- `git push origin <현재 브랜치>` 명령어로 현재 브랜치에 푸시합니다.
- 만약 사용자가 메시지 수정을 요청한 경우, 수정사항을 반영하여 **다시 승인을 받는 과정**을 반복합니다.

## 커밋 메시지 형식

```
<type>: <제목>
```

- **type**: `feat`, `fix`, `refactor`, `test`, `docs`, `chore` 중 적절한 것을 선택
- **제목**: 명령형 어조, 한글로 작성

## 사용자 확인 요청 형식

커밋을 수행하기 전, 아래의 형식으로 사용자에게 확인을 요청합니다.

```
## 커밋 대상 파일

- path/to/file1.js
- path/to/file2.js

## 제안 커밋 메시지

feat: 음식점 목록 컴포넌트 분리

## 푸시 대상 브랜치

lee-eojin

---

위 내용으로 커밋 및 푸시를 진행할까요? 수정이 필요하시면 말씀해주세요.
```

## 주의사항

- **[매우 중요]** 사용자의 **명시적 승인 없이는 절대로** `git add`, `git commit`, `git push`를 실행하지 않습니다.
- **[매우 중요]** 해당 스킬은 절대로 코드를 작성하거나 수정하지 않습니다. 오직 git 관련 동작만 수행합니다.
- 이미 staged된 파일이 있다면, 해당 내역을 사용자에게 먼저 보여주고 그대로 포함할지 질문합니다.
- 현재 브랜치가 `main`, `master`, `develop`, `dev` 등 보호 브랜치로 추정되는 경우, 푸시 전에 **반드시** 사용자에게 재확인을 받습니다.
- 원격 브랜치가 아직 존재하지 않는 경우 `git push -u origin <현재 브랜치>` 형태로 upstream을 설정하며 푸시합니다.
- push 과정에서 충돌이나 rejected 오류가 발생한 경우, 임의로 `--force` 옵션을 사용하지 않고 사용자에게 상황을 보고한 뒤 지시를 기다립니다.
